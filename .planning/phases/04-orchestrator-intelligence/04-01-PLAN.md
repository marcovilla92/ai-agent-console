    3. **src/runner/runner.py** -- Add `call_orchestrator_claude(prompt: str, schema: str) -> str` function. This is a NON-STREAMING call using `asyncio.create_subprocess_exec` with `proc.communicate()`. Command: `[claude, "-p", "--output-format", "json", "--json-schema", schema, "--dangerously-skip-permissions", prompt]`. Returns stdout decoded. Raises CalledProcessError on non-zero exit. Reuses `_resolve_claude()`.

    4. **src/agents/prompts/orchestrator_system.txt** -- Create the orchestrator system prompt with routing rules:
       - Normal forward flow: plan -> execute -> review
       - REVIEW DECISION "APPROVED" -> next_agent="approved"
       - REVIEW DECISION "BACK TO PLAN" -> next_agent="plan"
       - REVIEW DECISION "BACK TO EXECUTE" -> next_agent="execute"
       - For first pass (no prior iterations), always route forward
       - Be concise in reasoning (one sentence)

    5. **src/pipeline/orchestrator.py** -- Create the core module:
       - `OrchestratorDecision` dataclass: next_agent (str), reasoning (str), confidence (float), full_response (str), timestamp (auto-generated)
       - `OrchestratorState` dataclass (mutable): session_id, original_prompt, iteration_count=0, max_iterations=3, current_agent="plan", history=[], decisions=[], accumulated_handoffs=[], halted=False, approved=False
       - `ORCHESTRATOR_SCHEMA` -- JSON schema string for Claude CLI `--json-schema` flag with properties: next_agent (enum: plan/execute/review/approved), reasoning (string), confidence (number 0-1)
       - `build_orchestrator_prompt(state, latest_sections)` -- Builds prompt with: current agent name, iteration count, workflow history chain, truncated latest sections (500 chars max per section), asks "what agent should run next?"
       - `get_orchestrator_decision(state, latest_sections)` -- Calls `call_orchestrator_claude` with the built prompt and schema. Parses JSON response: extracts from `result` key (Claude CLI wraps in `{"result": {...}}`). Falls back to `parse_decision_from_text(raw)` if JSON fails.
       - `parse_decision_from_text(raw)` -- Regex/string fallback: looks for "APPROVED", "BACK TO PLAN", "BACK TO EXECUTE" in text. Returns OrchestratorDecision with confidence=0.3 on fallback.
       - `log_decision(db, session_id, decision, iteration_count)` -- Creates OrchestratorDecisionRecord and persists via OrchestratorDecisionRepository.
       - `orchestrate_pipeline(app, prompt, db, session_id)` -- The main async loop (see RESEARCH.md Pattern 2). Takes an `app` reference for status bar updates and panel streaming. Loop: run current agent via `stream_agent_to_panel`, record in history, increment iteration after review, call `get_orchestrator_decision`, update status bar, log to DB, handle decision (approved -> set approved, reroute -> check iteration limit then set current_agent, forward -> set current_agent). For now, reroute confirmation and halt dialog return stubs (True and "continue") -- Plan 02 wires real modals.
       - When building the prompt for a re-routed agent, prepend original prompt + all accumulated handoffs (from `state.accumulated_handoffs`) so each iteration builds on prior context.

    6. **tests/conftest.py** -- Add fixtures:
       - `mock_orchestrator_json_response` -- Returns valid JSON matching ORCHESTRATOR_SCHEMA
       - `orchestrator_state` -- Returns a pre-populated OrchestratorState for testing

    7. **tests/test_orchestrator.py** -- Tests for all behaviors listed above. Mock `call_orchestrator_claude` to return controlled JSON. Test build_orchestrator_prompt output format. Test parse_decision_from_text fallback. Test DB persistence.
